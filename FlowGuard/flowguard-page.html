01 — The Problem

Capability emergence is invisible to standard tools

Two agents. Both individually safe. Both approved. You deploy them together, and a new capability appears that neither held alone. This is capability emergence. Standard policy engines cannot see it because they reason about agents in isolation, not about what compositions of agents hold.

web-search-agent ✓ SAFE
  • base webSearch
  • forbidden codeExec, exfilData, networkAccess…
code-exec-agent
  • base codeExec, networkAccess, shellExec, webSearch
  • forbidden dataLeak, exfilData, remoteExec…

COMPOSITION FIRES TWO HYPEREDGES

webSearch ∧ codeExec → exfilData

We demonstrate that exfilData is an emergent property of the composition. While A (web) and B (exec) are individually compliant with the safety lattice, their union satisfies the preconditions for unauthorized data egress. This "hopwise" violation is caught by the Lean kernel, proving that safety is not preserved under composition.

codeExec ∧ networkAccess → remoteExec

In isolation, codeExec is a local sandbox risk and networkAccess is a data-flow risk. When composed, they synthesize a Remote Execution Primitive. This allows an agent to fetch and execute arbitrary payloads from reachable endpoints, bypassing local execution boundaries.

Emergent Capabilities Detected

Neither agent held exfilData or remoteExec individually. The composed system does. FlowGuard catches this. Standard policy engines do not.

02 — The Proofs

Lean kernel verified. Not tests, but certificates.

Every claim FlowGuard makes is backed by a formal proof accepted by the Lean 4 kernel. The kernel does not trust heuristics or test suites; it verifies mathematical derivations. 3304 compilation jobs passed with zero errors.

T-01
nonCompositionality_universal
CapHypergraph.lean
∀ (edges : List HyperEdge) (a b : Agent), isCapSafe edges a = true → isCapSafe edges b = true → (∃ e ∈ edges, ¬(e.premises ⊆ a.base) ∧ ¬(e.premises ⊆ b.base) ∧ e.premises ⊆ a.base ∪ b.base ∧ e.conclusion ∈ a.forbidden ∪ b.forbidden) → isCapSafe edges (compose a b) = false
For any agents satisfying the structural premises, individual safety does not survive composition. This is the headline result which is proved universally, not for a specific example.
T-02
capClosure_is_fixpoint
CapHypergraph.lean
∀ (edges : List HyperEdge) (S : Finset Cap), capClosure edges (capClosure edges S) = capClosure edges S
The capability closure operator is idempotent, which means applying it twice gives the same result as once. This guarantees FlowGuard computes the tightest possible emergent capability set, not an over-approximation.
T-03
cedar_nonCompositionality_gap
CedarBridge.lean
cedar_eval exfilData webAgent = CedarDecision.deny cedar_eval exfilData execAgent = CedarDecision.deny -- yet: isCapSafe edges (compose webAgent execAgent) = false
Cedar correctly denies exfiltration for every individual agent. However, the composed system can exfiltrate. This is a gap Cedar cannot close because it has no concept of capability hyperedge closure.
T-04
transitive_safety_misses_intermediate
InfoFlow.lean
∃ pipeline, isTransitivelySafe pipeline = true ∧ isHopwiseSafe pipeline = false
An endpoint-only information flow check can pass while an intermediate hop violates the security lattice. FlowGuard's per-hop check catches what the endpoint check misses.

FlowGuard reads the Python agent files directly — not documentation, not annotations. The AST (Abstract Syntax Tree) extractor walks the parsed code structure, inferring what each agent can and cannot do, and produces a formal capability spec. No API key required — this path runs entirely offline.

For richer inference, FlowGuard also supports an LLM extraction mode (currently using Gemini) which reads the full agent source and reasons about capabilities semantically. In the demo you are watching, AST mode is used. The --mode flag accepts ast, llm, or both. The latter runs AST first and lets LLM refine the result.

04 — Cedar's Structural Blind Spot

A correct tool answering the wrong question

Cedar, Amazon's formally verified policy language, is mathematically sound. It is not wrong. It is asking a question it was designed to ask: can this principal perform this action? That question is answered per-agent, per-request, in isolation. It cannot ask: what can this composition of agents hold?

✓ Cedar says

exfilData → DENY for every individual agent. Cedar evaluates each (principal, action) pair correctly. Its formal proof is sound. Lean verified this: cedar_eval exfilData webAgent = CedarDecision.deny.

Cedar is doing exactly what it was built to do. It checks whether a single agent, making a single request, is permitted to perform a single action. That question is answered correctly every time.

✗ FlowGuard says

The composed team CAN exfiltrate — via the hyperedge webSearch ∧ codeExec → exfilData. This is not a request Cedar can evaluate, because it is not a request at all. It is an emergent property of the combined capability set.

Cedar has no concept of hyperedge closure, it has no model of what two agents can do together. FlowGuard does. The gap is structural, not a bug in Cedar. Cedar would need to become a different kind of tool to close it.

This is not a criticism of Cedar. It is a proof that a fundamentally different formalism is required. Cedar reasons about requests. FlowGuard reasons about the emergent capability state of multi-agent systems. Both are necessary, and not substitutes of one another.
